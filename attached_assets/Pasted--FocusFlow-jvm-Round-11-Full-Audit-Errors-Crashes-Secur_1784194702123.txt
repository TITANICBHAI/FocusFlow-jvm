## Phase 2 — Deep review (code-review skill → architect subagent)
Prioritize files without a dedicated deep pass in rounds 1–10: `RegistryLockdown.kt`, `VpnBlocker.kt`, `WatchdogInstaller.kt`, `KillSwitchService.kt`, `GlobalKeyboardHook.kt`, `AppIconExtractor.kt`, `BlockPresets.kt`, `GlobalPin.kt`, `NuclearModePin.kt`, `DonateDialog.kt`, `EnforcementLog.kt`, `KeywordMatchLogger.kt`. Also diff everything changed since 2026-06-05 (`git log --since="2026-06-05" --stat`) — that's 6+ weeks outside what rounds 1–10 covered.

Specifically verify:
- **Taskbar-restore guarantee** — does it hold across every exit path: normal quit, tray quit, OS logoff/shutdown, and a hard crash where the shutdown hook itself never runs (e.g. power loss)? Is the startup-restore check in `Main.kt` early enough that no UI can block it first?
- **Dual-detection race** — can `WinEventHook` and the 500ms poll both fire on the same foreground app and both try to kill it? Any double-kill exception risk?
- **Cleanup completeness** — for `RegistryLockdown`, `NetworkBlocker` (firewall rules), and `HostsBlocker`: is every change guaranteed to be undone on disable/uninstall? What's left behind if the app is force-uninstalled mid-session without a clean disable first — orphaned firewall rules, hosts entries, registry keys?

## Phase 3 — Security & privacy-claim verification (new category — first-class, not an afterthought)

**Check this first:**
`CrashReporter.kt` and `ResourceMonitorService.kt` send crash reports / resource heartbeats to a Discord webhook, and `build-windows.yml` injects `FOCUSFLOW_WEBHOOK_CRASH` / `FOCUSFLOW_WEBHOOK_MONITOR` into *every* `packageExe` / `packageMsi` / `createDistributable` step — the actual shipped EXE/MSI/MSIX, not just dev builds. Meanwhile the website's homepage says **"0 — Data sent off your PC"** and "No internet required," and the Microsoft Store listing says **"No data collection."**

If those two GitHub secrets are set on the repo, the shipped app silently sends data off-device with no in-app disclosure or consent — contradicting both public claims, and potentially a Store-policy problem. Have the agent:
1. Flag this as an architectural risk regardless of whether the secrets are currently set — a future secret-add would silently start phoning home with no user-facing toggle.
2. Propose a fix: strip webhook telemetry from release builds entirely, *or* add an explicit in-app opt-in and disclose it in the privacy policy/store listing, *or* add a CI check that fails the build if webhook secrets are present on a public-release tag.
3. Report this clearly rather than silently picking a fix — it's a product decision, not just a bug.

**Also check:**
- Full-codebase grep for any remaining hardcoded secrets/tokens/webhook URLs beyond the two files round 9 already cleaned.
- Every `ProcessBuilder` / PowerShell / `cmd` invocation across the whole enforcement layer (not just `WatchdogInstaller`, already checked in round 9) — confirm array-form arguments, not string-concatenated shell commands, anywhere user-controlled input (app names, keywords, task titles) could reach a shell call.
- PIN hashing consistency across `SessionPin.kt`, `GlobalPin.kt`, `NuclearModePin.kt` — same algorithm/approach in all three? Does `PinHashingTest.kt` actually cover all three, or just one?
- Dependency CVE check: `sqlite-jdbc:3.47.1.0`, `jna` / `jna-platform:5.14.0`, `kotlinx-coroutines:1.7.3`.
- Behavior when firewall/registry operations run without admin rights — silent failure, or a clear error surfaced to the user?
- Auto-start registry entry (HKCU Run key) — cleaned up correctly on disable/uninstall, no dangling entry pointing at a deleted path.
