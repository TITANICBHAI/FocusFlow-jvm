AUDIT SCOPE — COVER ALL OF THESE
══════════════════════════════════════════════════════════

1. CONCURRENCY & THREAD SAFETY
   - Race conditions on shared mutable state between coroutines, 
     threads, and AWT/EDT
   - Missing @Volatile on fields written from one thread and read 
     from another
   - Non-atomic check-then-act sequences (TOCTOU)
   - Incorrect lock ordering that could deadlock (nested synchronized)
   - StateFlow / MutableStateFlow accessed without compareAndSet where 
     a guard is needed
   - coroutine Job references that are plain var (not @Volatile) but 
     written/read cross-thread

2. RESOURCE & LIFECYCLE MANAGEMENT
   - CoroutineScope never cancelled (scope leak after service stops)
   - Jobs not cancelled before a new one is launched (duplicate timers)
   - DB connections / cursors / streams not closed in all paths 
     (including exception paths)
   - AWT/Swing windows, frames, dialogs not disposed after use
   - ProcessBuilder processes whose streams are never read (blocks on 
     full pipe buffer)
   - JNA handles or native resources not freed

3. DATABASE CORRECTNESS
   - Missing transactions around multi-statement writes (partial 
     update leaves DB in inconsistent state)
   - Reads and writes mixed without isolation (dirty reads)
   - WAL-mode snapshot safety for backup/copy operations
   - Functions called on the UI/main thread instead of Dispatchers.IO
   - Silent exception swallowing that hides DB lock / busy errors

4. ENFORCEMENT CORRECTNESS
   - Process sets pushed to ProcessMonitor before or after the correct 
     service state is set
   - Blocked-process sets cleared in wrong order during session teardown
     (creates enforcement gap)
   - Schedule/allowance services not enforcing at startup (delayed 
     first tick)
   - Allowance time accumulation that drifts from real wall-clock time
   - Overnight schedule windows that drop enforcement at midnight
   - ProcessMonitor.sessionActive or alwaysOnEnabled set without 
     @Volatile or without the correct CAS guard

5. STATE MACHINE & TRANSITIONS
   - Services with enter/exit/start/stop that can be called twice 
     concurrently (missing guard)
   - State that is set but never cleared on the complementary 
     transition (dirty state on restart)
   - Callbacks or listeners registered but never unregistered (memory 
     / event leak)
   - Shutdown hooks registered multiple times

6. COROUTINE & ASYNC CORRECTNESS
   - blocking calls (JDBC, JNA, ProcessBuilder.waitFor, 
     Thread.sleep, System.in.read) inside coroutines without 
     Dispatchers.IO
   - delay(fixed_constant) accumulation loops where execution time 
     causes drift (use wall-clock delta instead)
   - runBlocking on the EDT or inside another coroutine
   - Coroutine launched on wrong dispatcher (UI state mutated on 
     Dispatchers.Default or IO)
   - withContext(Dispatchers.Main) inside @Synchronized — can 
     deadlock if caller already holds the monitor

7. DATA INTEGRITY
   - CSV / export functions that don't escape newlines, commas, or 
     quotes in field values (causes row/column misalignment)
   - Duplicate record creation when a "create if not exists" check 
     and the insert run without synchronization
   - Backup/restore operations that copy a live SQLite file without a 
     WAL checkpoint (produces a corrupt snapshot)

8. SECURITY & PRIVACY
   - PIN/auth gates that return "pass" when the stored hash is missing 
     due to a DB error (should deny, not silently allow)
   - Sensitive data (hashes, keys) written to logs or crash reports
   - Enforcement bypass: any code path that disables blocking without 
     the required PIN / session check

9. STARTUP & SHUTDOWN SEQUENCING
   - Services started before their dependencies are ready 
     (creates an enforcement window)
   - dispose() / stop() not called for every started service in the 
     shutdown sequence
   - Daemon threads used where non-daemon is required to guarantee 
     OS-level cleanup runs before JVM exit

10. SILENT FAILURE SURFACES
    - Empty catch blocks around critical paths (DB opens, JNA calls, 
      network rules) that leave the app in a degraded state with no 
      indication to the user or logs
    - Fallback values that mask real errors 
      (e.g., returning "pass" when a null means "error")
